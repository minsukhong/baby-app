## 🔴 반드시 지킬 규칙

### 0. 코드 수정 후 **무조건** 아래 순서 실행 (매번 예외 없음)

```
1. 파일 수정
2. HMR 컴파일 확인 (3~5초 대기 후 로그 체크)
3. 컴파일 안 됐으면 → 서버 재시작 (touch는 신뢰 못함 — 규칙 8 참조)
4. 포트 3000 확인 필수 (3001/3002/3003이면 구 프로세스 살아있는 것)
5. 브라우저 강력 새로고침 (Ctrl+Shift+R)
```

**HMR 컴파일 확인 명령어** (수정 후 반드시 실행):
```bash
# ① 수정 전 — 현재 컴파일 횟수 기록
docker exec baby-app_devcontainer-frontend-1 sh -c \
  "strings /tmp/next.log | grep -c 'Compiled'"

# ② 파일 수정

# ③ 수정 후 5초 대기 — 컴파일 횟수가 늘었는지 확인
sleep 5 && docker exec baby-app_devcontainer-frontend-1 sh -c \
  "strings /tmp/next.log | grep -c 'Compiled'"
```
→ ③의 숫자가 ①보다 커야 새로 컴파일된 것. **같으면 즉시 서버 재시작.**
→ ⚠️ `tail -3`로 로그 마지막 줄만 보면 기존 줄과 새 줄을 구분 못함 — **반드시 횟수(count)로 비교**

**안전 재시작 명령** (kill → 확인 → 시작 순서 지킬 것):
```bash
# Step 1: 모든 next/node 프로세스 kill
docker exec baby-app_devcontainer-frontend-1 sh -c \
  "ps aux | grep -E 'next|npm|node' | grep -v 'grep|vscode|tsserver|eslint|tailwind' \
   | awk '{print \$1}' | while read pid; do kill -9 \$pid 2>/dev/null; done; echo done"

# Step 2: ⚠️ kill 후 반드시 0개인지 확인 — 남아있으면 Step 1 재실행
sleep 2 && docker exec baby-app_devcontainer-frontend-1 sh -c \
  "ps aux | grep -E 'next-server' | grep -v grep | wc -l"
# → 0 이어야 함. 1 이상이면 Step 1 다시 실행

# Step 3: 0개 확인 후에만 새 서버 시작
docker exec -d baby-app_devcontainer-frontend-1 sh -c "cd /app && npm run dev > /tmp/next.log 2>&1" && \
sleep 10 && \
docker exec baby-app_devcontainer-frontend-1 sh -c "strings /tmp/next.log | grep -E 'Ready|localhost'"
```
→ 반드시 `localhost:3000` 확인. 3001 이상이면 Step 1~2 재실행.

### 1. import 중복 금지 — 파일 수정 시 기존 import 먼저 확인

**실수**: 이미 있는 import를 또 추가해서 TypeScript 컴파일 에러 발생.

**예방**: 파일 수정 전 `Read` 도구로 import 섹션 반드시 확인. 같은 모듈에서 여러 개 import 시 한 줄로 합칠 것.
```typescript
// ❌ 잘못된 예
import { getMyProfile } from "@/lib/profile";
import { ProfileData } from "@/lib/profile";

// ✅ 올바른 예
import { getMyProfile, ProfileData } from "@/lib/profile";
```

### 2. 서버 재시작 시 포트 3000 확인 필수

**실수**: 재시작 명령을 여러 번 실행하면서 구 프로세스를 kill하지 않고 새 인스턴스를 계속 추가 기동 → 프로세스 4개 누적, 포트 3003까지 올라감.

**증상**: 코드를 바꿨는데 화면이 그대로 → 유저가 "안 바뀌는데?"

**올바른 절차**: 규칙 0의 안전 재시작 명령 참조.

**주의**: 포트가 3001, 3002, 3003으로 뜨면 구 프로세스가 살아있는 것 → kill 후 재시작

### 2-2. Next.js .next 폴더 삭제는 신중하게

**실수**: `.next/cache`만 삭제했는데 `.next` 전체를 삭제하거나, 삭제 후 빌드 없이 dev 서버를 올림.

**증상**: JS 청크 404 → "혜택 계산 중..." 로딩 무한

**올바른 절차**:
```bash
# .next 전체 삭제 시 반드시 빌드 먼저
docker exec frontend-1 sh -c "cd /app && rm -rf .next && npm run build 2>&1 | tail -5"
# 빌드 성공 확인 후 dev 서버 시작
```

### 3. 로컬 DB ≠ Supabase — 반드시 같은 DB 바라보게 할 것

**실수**: 로컬 Docker DB를 바라보도록 설정되어 있어 데이터 없음 → 총 혜택 0원.

**현재 확정된 구조**:
- 로컬 백엔드 → **Supabase** (`.devcontainer/docker-compose.yml`에 DATABASE_URL 설정됨)
- Railway 백엔드 → **Supabase** (Railway 환경변수)
- 로컬 Docker DB (`db` 컨테이너)는 **사용하지 않음**

**확인 방법**:
```bash
docker exec backend-1 sh -c "env | grep DATABASE_URL"
# postgresql://postgres.rgcfofduvijaovwijszy:... 가 나와야 정상
```

### 4. benefits 필터링은 `applies_to_pregnant` / `applies_to_born` 플래그 기준

**실수**: stage 컬럼("birth", "infant", "anytime" 등)만으로 잠금 처리 → `anytime` stage인 아동수당이 임신 중인데 "신청가능"으로 표시됨.

**올바른 로직** (`determine_status` 함수):
```python
# applies_to 플래그가 stage보다 정확 — 반드시 먼저 체크
if is_pregnant and not benefit.applies_to_pregnant:
    return "locked", "출산 후 신청 가능해요"
if not is_pregnant and not benefit.applies_to_born:
    return "locked", "임신 중에만 해당해요"
```

### 5. 금액 표시: 헤더는 정확한 원단위, 카드는 압축 표기

**실수**: `formatAmount(11_850_000)` → `1천만원` (toFixed(0)으로 반올림됨)

**올바른 방식**:
- 헤더(메인 총액): `formatAmountFull()` → `11,850,000원`
- 카드/서브 금액: `formatAmount()` → `1,085만원`

```typescript
// lib/benefits.ts
export function formatAmountFull(amount: number): string {
  return amount.toLocaleString("ko-KR") + "원";
}
export function formatAmount(amount: number): string {
  if (amount >= 10_000_000) return `${(amount / 10_000).toLocaleString("ko-KR")}만원`;
  ...
}
```

### 7. 총 혜택 금액은 확정금액 + 조건부금액 분리 표시

**실수**: 소득조건 있는 혜택까지 합산해서 하나의 숫자로 표시 → 사용자가 실제로 받지 못할 수도 있는 금액이 포함됨.

**올바른 방식**:
- `confirmed_amount`: 소득조건 없는 혜택 합계 → 메인 표시
- `conditional_amount`: 소득조건 있는 혜택 합계 → `+α 300만원` 형식 별도 표시
- 달성률 기준 = `confirmed_amount`

### 8. Write/Edit 도구로 컴포넌트 파일 수정 시 HMR이 자동 트리거 안 될 수 있음

**실수**: BenefitCard.tsx를 Write 도구로 수정했는데 HMR이 트리거되지 않아 유저에게 구버전이 서빙됨.

**잘못된 확인 방법**:
```bash
# ❌ tail -3 으로만 보면 기존 컴파일 줄과 새 컴파일 줄 구분 불가
strings /tmp/next.log | grep 'Compiled' | tail -3
# → 서버 시작 때 찍힌 "Compiled /dashboard" 가 그대로 나와도 새 컴파일처럼 보임
```

**올바른 확인 방법**: 수정 전/후 횟수(count) 비교 (규칙 0 참조).

**예방**: Write/Edit 도구로 프론트 파일 수정 후 5초 내 count 증가 없으면 즉시 서버 재시작.

### 8-2. `touch` 명령으로 HMR 강제 트리거 불가

**실수**: 컴파일이 안 됐을 때 `touch /app/app/dashboard/page.tsx`로 HMR을 강제 트리거하려 했지만 작동 안 함.

**증상**: touch 후에도 로그에 새 컴파일 이벤트가 찍히지 않음.

**올바른 방법**: HMR 미반영 시 서버 전체 재시작 (규칙 0의 안전 재시작 명령 사용).

### 9. kill 후 프로세스 잔존 확인 없이 새 서버 시작 → 포트 충돌

**실수**: kill 명령 실행 후 프로세스가 실제로 0개인지 확인 안 하고 바로 새 서버 시작 → 남은 프로세스가 3000 점유 → 새 서버가 3002로 뜸.

**올바른 순서**: kill → `ps aux | grep next-server | grep -v grep | wc -l` 로 0 확인 → 그다음에 새 서버 시작 (규칙 0 안전 재시작 참조).

### 9-2. `xargs kill -9` 패턴은 Exit code 123 에러 발생 가능

**실수**: `ps aux | grep | awk | xargs kill -9` 패턴 사용 → grep 결과가 없거나 공백일 때 `Exit code 123` 에러로 이후 명령이 실행 안 됨.

**올바른 패턴**: `xargs` 대신 `while read` 루프 사용:
```bash
ps aux | grep -E 'next-server|next dev|npm run dev' | grep -v grep \
  | awk '{print $1}' | while read pid; do kill -9 $pid 2>/dev/null; done; echo done
```
→ 프로세스가 없어도 에러 없이 `done` 출력 후 정상 종료.

### 10. 그라데이션 배경 위 버튼 색상 대비 부족

**실수**: 보라색 그라데이션 헤더 위에 `bg-white/20 text-white` 버튼 → 배경과 구분이 안 돼 유저가 버튼을 찾지 못함.

**올바른 방법**: 그라데이션/진한 배경 위 버튼은 충분한 대비 확보:
```tsx
// ❌ 잘 안 보임
className="text-xs bg-white/20 px-3 py-1.5 rounded-xl"

// ✅ 명확하게 보임
className="text-xs bg-white/30 border border-white/50 px-3 py-1.5 rounded-xl font-medium"
```

### 11. CSS 3D 플립 잔상 — backface-visibility webkit 접두사 누락

**실수**: `backfaceVisibility: "hidden"` 만 설정 → Safari/Chrome에서 뒤집을 때 앞면 그라데이션이 잔상으로 비침.

**올바른 방법**: webkit 접두사 반드시 함께 설정:
```tsx
style={{
  backfaceVisibility: "hidden",
  WebkitBackfaceVisibility: "hidden",   // ← 필수
  transformStyle: "preserve-3d",
  WebkitTransformStyle: "preserve-3d",  // ← 필수
}}
```

### 12. CSS animation을 3D 회전 부모에 걸면 자식에게도 함께 회전 적용됨

**실수**: 카드 테두리 box-shadow 애니메이션(`border-glow`)을 회전하지 않는 외부 래퍼에 적용 → 카드가 돌아가도 테두리 glow는 제자리에 남아 잔상처럼 보임.

**올바른 방법**: box-shadow 애니메이션은 `rotateY` transform을 가진 플립 컨테이너에 직접 적용 → 카드와 함께 회전하면서 자연스럽게 사라짐.
```tsx
// ❌ 외부 래퍼에 걸면 카드와 별개로 움직임
<div style={{ animation: "border-glow ..." }}>
  <div style={{ transformStyle: "preserve-3d", transform: "rotateY(...)" }}>

// ✅ 회전 컨테이너에 직접 걸면 함께 회전
<div style={{
  transformStyle: "preserve-3d",
  transform: "rotateY(...)",
  animation: "border-glow ...",
}}>
```

## 📋 코드 수정 후 체크리스트

```
[ ] 백엔드 수정 → uvicorn auto-reload 로그 확인
[ ] 프론트 수정 → Next.js HMR 컴파일 확인 (3000 포트)
[ ] HMR 확인 → 수정 전/후 Compiled count 비교 (규칙 8) — tail -3 금지
[ ] count 안 늘었으면 즉시 서버 재시작 (touch 금지 — 규칙 8-2)
[ ] kill → next-server 0개 확인 → 그다음 새 서버 시작 (규칙 9)
[ ] 포트 확인 → localhost:3000 (3001 이상이면 다시 kill)
[ ] UI 버튼 색상 → 배경 대비 충분한지 확인 (규칙 10)
[ ] API 응답 확인 → curl 또는 브라우저 Network 탭
[ ] 브라우저 강력 새로고침 (Ctrl+Shift+R)
```
