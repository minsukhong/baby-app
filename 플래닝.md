# baby-app 플래닝
> 목적: 개발 시작 전 전체 설계를 확정하는 문서.
> 규칙: 이 문서가 확정되기 전까지 코드 작업 금지. 변경 시 반드시 개발자 승인 후 반영.

---

## 서비스 정의

### 한 줄 정의
> "흩어진 출산·육아 혜택을 내 상황에 맞게 모아주고, 검증된 육아용품을 가장 싸게 살 수 있게 도와주는 앱"

### 핵심 가치
1. **정확한 맞춤 혜택** → 내 프로필(지역/소득/출산형태)에 맞는 혜택만 보여줌
2. **신뢰할 수 있는 육아용품** → 실제 엄마들이 검증한 필수품 큐레이션
3. **최저가 구매 경로** → 쿠팡 / 네이버쇼핑 / 중고마켓 연동
4. **심플한 UX** → 피곤한 부모가 한 손으로도 쉽게 사용

### 수익 모델
- 쿠팡 파트너스 링크 (최저가 연동 시 자동 수익)
- 육아용품 한정 타겟 광고 (스팸 없이, 좋은 제품만)

## 데이터 전략

### 핵심 원칙
```
데이터 품질 = 앱의 신뢰 = 사용자 유지
```
- 내가(AI) 조사 → 출처 + 수집 이유 명시 → 개발자 승인 → DB 등록
- 승인되지 않은 데이터는 절대 앱에 노출하지 않음
- 변경 감지 시 재보고 → 재승인 → 업데이트

### 데이터 종류별 성격

| | 혜택 데이터 | 육아용품 데이터 |
|--|------------|---------------|
| 출처 | 정부/지자체 (공식) | 실사용자/커뮤니티 (비공식) |
| 업데이트 주기 | 연 1~2회 (예산 시즌) | 수시 (트렌드 변화) |
| 검증 기준 | 공문/고시 정확성 | 실사용 후기 신뢰도 |
| 핵심 리스크 | 금액/조건 오류 | 트렌드 뒤처짐 |

## AI 에이전트 아키텍처

### 전체 흐름

```
[외부 소스]                    [외부 소스]
정부/지자체 사이트              맘카페/커뮤니티/쿠팡
      ↓                              ↓
[Agent 1]                      [Agent 2]
혜택 수집 에이전트              육아용품 수집 에이전트
      ↓                              ↓
      └──────────┬───────────────────┘
                 ↓
          [Agent 3: 검증]
          교차 검증 + 신뢰도 점수
                 ↓
          [개발자 승인]
          승인 → DB 등록
          반려 → 재조사
                 ↓
          [혜택 DB / 육아용품 DB]
                 ↓
          [Agent 4: 개인화] ← 실시간
          사용자 프로필 매칭
                 ↓
          [사용자 앱 화면]
```

---

### Agent 1: 혜택 수집 에이전트

```
역할    : 전국 출산/육아 혜택 데이터 수집
탐색처  : 복지로, 정부24, 공공데이터포털,
          각 지자체 홈페이지 (서울시, 구청 등)
          아이사랑 포털, 고용보험 사이트
출력물  : 혜택 보고서
          - 혜택명
          - 지원 내용 및 금액
          - 신청 조건 (소득, 지역, 출산형태 등)
          - 출처 URL
          - 수집 이유 ("단태아/다태아 금액 차이 있음" 등)
          - 마지막 업데이트 날짜
실행 시점: 수동 (요청 시) + 자동 (월 1회 변경 감지)
기술    : Claude API + Brave Search API + BeautifulSoup
```

### Agent 2: 육아용품 수집 에이전트

```
역할    : 실사용자 검증 육아 필수품 데이터 수집
탐색처  : 네이버 맘카페 (아이엔젤, 쌍둥이맘 등)
          육아 유튜버 추천 목록
          쿠팡 육아용품 베스트셀러
          맘 인플루언서 리스트
출력물  : 육아용품 보고서
          - 제품명 / 카테고리 / 월령
          - 추천 이유 ("신생아 필수 1위", "쌍둥이맘 필수" 등)
          - 출처 URL (커뮤니티 링크)
          - 언급 빈도 (얼마나 많이 추천됐는지)
          - 주의사항 ("국산만 선호하는 경향" 등)
          - 마지막 업데이트 날짜
실행 시점: 수동 (요청 시) + 자동 (분기 1회)
기술    : Claude API + Brave Search API + BeautifulSoup
```

### Agent 3: 검증 에이전트

```
역할    : Agent 1, 2 결과물 신뢰도 검증
방식    : 복수 출처 교차 확인
          정부 공문 vs 뉴스기사 vs 커뮤니티 일치 여부
출력물  : 신뢰도 점수 (높음 / 보통 / 낮음 / 재확인 필요)
          + 불일치 항목 표시
실행 시점: Agent 1, 2 실행 직후 자동 실행
기술    : Claude API
```

### Agent 4: 개인화 에이전트

```
역할    : 사용자 프로필 기반 맞춤 데이터 필터링
입력    : 사용자 프로필 (출산형태, 지역, 소득, 직업 등)
출력    : 해당 사용자에게 맞는 혜택 목록
          해당 사용자 월령에 맞는 육아용품 목록
실행 시점: 사용자 앱 접속 시 실시간
기술    : Claude API + PostgreSQL 쿼리
```

### 기술 스택 요약

| 항목 | 기술 |
|------|------|
| AI 모델 | Claude claude-sonnet-4-6 (Anthropic API) |
| 웹 검색 | Brave Search API (무료 2천건/월) |
| 크롤링 | BeautifulSoup + requests |
| 스케줄러 | GitHub Actions (월 1회 자동 실행) |
| 에이전트 프레임워크 | Anthropic Agent SDK |
| 보고서 저장 | Markdown 파일 → 개발자 검토 후 DB 반영 |

## DB 설계 (초안)

### 혜택 테이블
```
benefits
├── id
├── name              혜택명
├── stage             단계 (pregnant / newborn / infant / work_life)
├── category          카테고리 (medical / cash / service / leave)
├── description       상세 내용
├── amount_single     단태아 금액
├── amount_multiple   다태아 금액
├── amount_first      첫째 금액
├── amount_second     둘째 이상 금액
├── region_scope      적용 지역 (전국 / 서울 / 송파구 등)
├── condition_income  소득 조건 (없음 / 구간)
├── condition_job     직업 조건 (전체 / 직장인 / 자영업)
├── apply_url         신청 URL
├── source_url        출처 URL
├── verified_at       검증 날짜
├── is_active         노출 여부
└── updated_at
```

### 육아용품 테이블
```
baby_products
├── id
├── name              제품명
├── category          카테고리 (침구 / 수유 / 목욕 / 외출 등)
├── age_range         월령 (0-3 / 3-6 / 6-12 / 12+)
├── rank              인기 순위
├── why_recommended   추천 이유
├── source_url        출처 (커뮤니티 링크)
├── coupang_url       쿠팡 링크
├── naver_url         네이버쇼핑 링크
├── is_twin_special   쌍둥이 특화 여부
├── verified_at       검증 날짜
├── is_active         노출 여부
└── updated_at
```

## 혜택 센터 완성 현황 (2026-03)

### 온보딩 (4단계)
| 단계 | 내용 |
|------|------|
| Step 1 | 임신중/출산함 + 예정일/생일 (연월일 드롭다운) |
| Step 2 | 기존 자녀수 + 이번 아이 수 (총 N명 안내) |
| Step 3 | 거주 지역 + 직업 상태 |
| Step 4 | 소득 구간 (선택, 모르면 스킵 가능) |

### 대시보드
- 상단 요약: 총 혜택금액(원단위 정확 표기) + 소득조건부 별도(+α)
- 달성률 바: 확정금액 기준 100%, 초과 시 "🎉 달성 초과!"
- 타임라인: 출산 전 / 출산 후 섹션 구분
- 혜택 카드: 금액 내림차순, 소득조건 혜택 맨 뒤
- 배지: "누구나 신청"(초록), "소득조건"(주황), 지역명(파랑)
- 상태 버튼: 신청했어요 → 수령완료, 각 단계별 취소 가능
- 프로필 수정 버튼: 헤더 우상단 (대비 확보된 디자인)

### 혜택 카드 모달 (BenefitActionModal)
- 신청했어요 / 수령완료 / 상세보기 클릭 시 팝업 모달 오픈
- 3D 카드 플립 (앞면 ↔ 뒷면), 동일 위치 🔄 버튼으로 일관성 유지
- 앞면: 카테고리 그라데이션 헤더, 금액 글로우 박스, 배지, 신청 CTA
- 뒷면: 상세 정보 (신청기한, 소득조건, 지급방식 등), 공식사이트 링크
- 테두리: box-shadow 색상 순환 (보라→핑크→초록), 카드와 함께 회전
- 신청 확정 시: 카드 중앙에서 360도 꽃가루 버스트 → 1.8초 후 자동 닫힘

### 혜택 금액 개인화 로직 (4가지 타입)
| 타입 | 조건 | 계산 방식 |
|------|------|----------|
| 1 | amount_single / amount_multiple 있음 | 다태아 여부로 단건 선택 |
| 2 | amount_first / amount_second_plus 있음 | 자녀별 출생순서로 합산 |
| 3 | is_per_child = True | representative_amount × 현재 아이수 |
| 4 | 그 외 | representative_amount 고정 |

### 혜택 필터링 로직
| 조건 | 처리 |
|------|------|
| 임신/출산 여부 | `applies_to_pregnant` / `applies_to_born` 플래그 |
| 임신 주차 | `application_start_week` vs 현재 주차 |
| 신청 기한 | `apply_within_days` vs 출산 후 경과일 |
| 직업 | `applies_to_employed/self_employed/unemployed` |
| 지역 | `region` vs 프로필 지역 (REGION_MAP 정규화) |
| 소득 | `income_max_ratio` vs `income_range` |
